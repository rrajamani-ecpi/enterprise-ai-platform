# Feature Specification: Image Generation & Vision

**Feature Branch**: `022-image-generation-vision`

**Created**: 2026-07-21

**Status**: Draft

**Input**: Derived from `docs/PRODUCT_REQUIREMENTS_DOCUMENT.md` §4.14 (Image Generation & Vision), REQ-IMG-1 through REQ-IMG-3. This spec covers the image-generation and vision-input CAPABILITY itself — prompt-to-image generation, output validation/storage/rendering, and passing user-supplied images to vision-capable models. It does NOT own the cross-cutting tool-gating/availability mechanism that determines which callers/roles may invoke the ImageGen tool at all — that mechanism is owned by `specs/020-tools-extensions-framework/spec.md` (see Assumptions), which lists ImageGen in its minimum tool catalog. Pre-flight size-limit rejection for user-supplied image attachments (>5MB) is an existing requirement of `specs/004-chat-message-pipeline/spec.md` (FR-022) and is cross-referenced, not duplicated, here.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Generate an image from a prompt and see it in chat (Priority: P1)

A user asks a persona to generate an image from a text prompt. The system invokes an image-generation capability, stores the resulting output in object storage, and renders it inline in the chat thread as a persistent, reloadable artifact rather than an ephemeral response.

**Why this priority**: This is the core capability PRD §4.14 describes (REQ-IMG-1) — without it, there is nothing for validation or vision-input handling to sit on top of. A generated image that isn't durably stored and renderable on reload is not a shippable feature.

**Independent Test**: Prompt a persona to generate an image, confirm it renders in the chat thread, reload the thread, and confirm the same image still renders from stored blob storage (not from an ephemeral/expired URL).

**Acceptance Scenarios**:

1. **Given** a user submits a prompt requesting an image, **When** the image-generation tool completes successfully, **Then** the output is written to object storage and a reference to it (not the raw bytes) is persisted on the chat message.
2. **Given** a chat message referencing a previously generated image, **When** the thread is reloaded, **Then** the image renders identically from its stored reference.
3. **Given** an image-generation request fails at the provider (timeout, content-policy rejection, quota), **When** the failure occurs, **Then** the user sees a clear failure indication in the thread and no broken/dangling storage reference is persisted.

---

### User Story 2 - Generated and received images are validated before storage or use (Priority: P1)

Before an image (generated by the tool, or otherwise received into the pipeline) is written to storage or handed to a model, the system checks its type and size against allow-listed constraints, rejecting anything that doesn't conform.

**Why this priority**: PRD REQ-IMG-2 requires validation as a precondition of storage/use, not an afterthought — an unvalidated image (wrong MIME type, oversized, malformed) reaching blob storage or a model call is both a reliability and a security exposure. Ranked P1 alongside Story 1 because generation without validation is not a safe MVP.

**Independent Test**: Attempt to store a generated or received image with a disallowed MIME type, and separately one exceeding the configured size ceiling, and confirm both are rejected before any write to blob storage or any model call.

**Acceptance Scenarios**:

1. **Given** an image produced by the image-generation tool, **When** it is about to be stored, **Then** its content type is checked against an allow-list (e.g., PNG/JPEG/WebP) and its size against a configured maximum before the write occurs.
2. **Given** a generated image that fails type or size validation, **When** validation runs, **Then** the image is rejected, not stored, and not referenced in the chat message.
3. **Given** a user-supplied image attachment intended for a vision-capable model, **When** it exceeds the provider's size limit, **Then** it is rejected pre-flight per the existing check already specified in `specs/004-chat-message-pipeline/spec.md` (FR-022, >5MB for Anthropic) — this spec does not re-specify that threshold or its enforcement point, only that the same validate-before-use principle applies uniformly to generated images.

---

### User Story 3 - User-supplied images are passed to vision-capable models (Priority: P2)

A user attaches an image to a chat message intended for a persona backed by a multimodal model. The system passes the (validated) image as an input to the model so it can reason over the image content, but only when the resolved model actually supports vision.

**Why this priority**: PRD REQ-IMG-3 depends on both attachment validation (Story 2 / spec 004) and on knowing whether the resolved model supports vision at all — routing an image to a non-vision model is a defined failure mode, not an open question. Ranked P2: it extends the chat pipeline's existing multimodal handling rather than introducing new storage/generation surface.

**Independent Test**: Attach a validated image to a message routed to a model whose catalog entry has the vision capability flag set, and confirm the model receives the image as an input; separately, route the same attachment to a model without the vision flag and confirm a defined rejection/fallback rather than a silent drop or provider-level failure.

**Acceptance Scenarios**:

1. **Given** a user-supplied image attachment that has passed pre-flight validation, **When** the active model's catalog entry has the vision capability flag set (per `specs/014-model-access-config-management/spec.md` FR-011), **Then** the image is included as an input to the model call.
2. **Given** a user-supplied image attachment, **When** the active model's catalog entry does not have the vision capability flag set, **Then** the system does not silently drop the image or fail at the provider — it surfaces a defined, typed rejection or routes to a vision-capable default per the model-substitution behavior already specified in `specs/004-chat-message-pipeline/spec.md` (FR-020/FR-021), whichever this platform's existing substitution behavior dictates.
3. **Given** an image passed to a vision-capable model, **When** the model responds, **Then** the response is treated as ordinary assistant output in the thread (no special-cased rendering divergent from text responses).

### Edge Cases

- What happens when image generation succeeds at the provider but the subsequent blob-storage write fails (partial success) — is the user shown a failure, and is the tool's structured `{success:false}` failure shape (per spec 004, FR-011) used here as well?
- How does the system handle a prompt that violates the image-generation provider's content policy — is this surfaced as a distinct error from a generic tool failure?
- What happens when a user attaches multiple images to a single message intended for a vision model — is there a per-message image count limit?
- How does validation handle an image whose file extension and actual content-type/magic-bytes disagree (e.g., a `.png`-named file containing non-image bytes)?
- What happens to a stored generated image's blob if the referencing chat message or thread is later deleted — is it garbage-collected, orphaned, or retained under a separate retention policy?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST support generating an image from a user-supplied text prompt via an image-generation tool invocation.
- **FR-002**: Successfully generated images MUST be written to object (blob) storage, with the chat message persisting a reference to the stored object rather than embedding raw image bytes.
- **FR-003**: A chat message referencing a generated image MUST render that image identically on thread reload, sourced from the persisted storage reference.
- **FR-004**: A failed image-generation attempt (provider error, timeout, content-policy rejection) MUST surface a clear failure to the user and MUST NOT persist a dangling or partial storage reference.
- **FR-005**: Every image (tool-generated or user-received) MUST be validated for content type against an allow-list and for size against a configured maximum before it is written to storage or passed to a model.
- **FR-006**: An image failing type or size validation MUST be rejected — not stored and not referenced in any chat message.
- **FR-007**: Pre-flight size validation for user-supplied image attachments destined for a model call is governed by the existing check in `specs/004-chat-message-pipeline/spec.md` (FR-022); this spec does not duplicate or override that threshold.
- **FR-008**: The system MUST pass a validated user-supplied image attachment as an input to the resolved model's call only when that model's catalog entry (per `specs/014-model-access-config-management/spec.md` FR-011) has the vision capability flag set.
- **FR-009**: WHEN a user-supplied image attachment is present but the resolved model does not have the vision capability flag set, THE SYSTEM MUST NOT silently drop the image or fail at the provider — it MUST apply a defined, typed rejection or the existing model-substitution behavior (`specs/004-chat-message-pipeline/spec.md` FR-020/FR-021).
- **FR-010**: Access to and availability of the image-generation tool itself (which callers/roles may invoke it) is governed by the tool-gating framework owned by `specs/020-tools-extensions-framework/spec.md`; this spec assumes that gate is already resolved before generation begins and does not re-specify it.

### Key Entities *(include if feature involves data)*

- **GeneratedImage**: a blob-storage object produced by the image-generation tool — content type, size, storage reference/URI, and the prompt that produced it. Referenced (not embedded) from the owning chat message.
- **ImageAttachment**: a user-supplied image submitted with a chat message for vision-model input; subject to the pre-flight size check owned by spec 004 and the type/size validation owned by this spec before being passed to a model.
- **ModelConfigDocument (vision flag)**: the existing catalog entry (spec 014) whose vision capability flag gates whether an `ImageAttachment` may be passed to a given model's call.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of successfully generated images are retrievable and render identically after a thread reload, across a test corpus of generation requests.
- **SC-002**: 0 generated or received images bypass type/size validation before reaching blob storage or a model call, across a test corpus including at least one disallowed-type and one oversized case.
- **SC-003**: 100% of failed image-generation attempts leave no dangling/partial storage reference on the chat message.
- **SC-004**: 100% of validated image attachments routed to a vision-flagged model are included as model input; 100% routed to a non-vision-flagged model receive the defined rejection/substitution behavior instead of a silent drop or provider-level error.

## Assumptions

- **Cross-reference to spec 020**: `specs/020-tools-extensions-framework/spec.md` (drafted concurrently) owns the cross-cutting tool-gating/availability mechanism and lists ImageGen in its minimum tool catalog. This spec assumes that framework governs whether a caller/role may invoke the image-generation tool at all (FR-010) and does not re-specify gating logic here. If 020 has not yet landed at implementation time, this note is the forward pointer to reconcile against once it does.
- **Cross-reference to spec 004**: pre-flight size rejection of user-supplied image attachments (>5MB for Anthropic, 400 `IMAGE_TOO_LARGE`) is already specified in `specs/004-chat-message-pipeline/spec.md` (FR-022, User Story 5). This spec reuses that check rather than duplicating its threshold, and adds the type/size validation for tool-*generated* images as new scope (FR-005/FR-006).
- **Cross-reference to spec 014**: whether a given model supports vision input is governed by the capability-flag catalog already specified in `specs/014-model-access-config-management/spec.md` (FR-011). This spec consumes that flag (FR-008/FR-009) rather than re-specifying model capability metadata.
- **Storage backend**: per Constitution Principle I (Azure-Only, No AWS Vestiges), generated images are stored in Azure Blob Storage; no S3-shaped storage code or AWS SDK usage may be introduced for this capability.
- The image-generation tool's own failure handling (structured `{success:false}` payload, no raw exception propagation) reuses the tool-failure envelope already specified in `specs/004-chat-message-pipeline/spec.md` (FR-011, User Story 4) rather than introducing a new one.
- Retention/lifecycle policy for generated-image blobs (e.g., garbage collection on thread/message deletion) is flagged as an open edge case and is not decided by this spec.
